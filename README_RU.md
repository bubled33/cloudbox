## Введение

**Проект**: Облачное хранилище файлов (Google Drive Lite)

**Цель**: Разработка веб‑сервиса для хранения, управления и обмена файлами с поддержкой версионирования и генерации превью.

| Показатель                  | Символ           | Требование            |
| --------------------------- | ---------------- | ---------------------- |
| Доступность сервиса         | $Uptime$         | $Uptime \geq 99.5\%$   |
| Время восстановления (RTO)  | $RTO$            | $RTO \leq 1 \text{ h}$ |
| Точка восстановления (RPO)  | $RPO$            | $RPO \leq 1 \text{ h}$ |
| Задержка API (p95)         | $Latency_{p95}$  | $\leq 200 \text{ ms}$  |
| Пропускная способность (загрузка) | $RPS_{upload}$   | $\leq 100$             |
| Пропускная способность (скачивание) | $RPS_{download}$ | $\leq 300$             |
| Конкурентные пользователи   | $N_{active}$     | $\geq 1000$            |
| Максимальный размер файла   | $Size_{file}$    | $\leq 10 \text{ GB}$   |
| Общая ёмкость системы (MVP) | $Size_{total}$   | $\geq 1 \text{ TB}$    |
| Время обработки превью      | $T_{preview}$    | $\leq 30 \text{ s}$    |
| Ёмкость очереди             | $N_{queue}$      | $\leq 10^{4}$          |
| Успешность задач            | $P_{success}$    | $\geq 99\%$            |
| TTL публичной ссылки        | $TTL_{link}$     | $\leq 600 \text{ s}$   |

## Технологический стек

Система реализована на **современном Go‑стеке**, объединяющем высокопроизводительные веб‑фреймворки, надежные базы данных, масштабируемые очереди и фоновые воркеры.

| Слой / Назначение            | Технология / Библиотека                     | Примечания / Использование |
|------------------------------|-----------------------------------------------|---------------------------|
| **Язык программирования**    | Go (Golang)                                   | Основной бэкенд, компилируется в единый бинарник, подходит для высокопроизводительных сервисов |
| **Веб‑фреймворк / Роутер**   | [Gin](https://github.com/gin-gonic/gin)       | Высокопроизводительный HTTP фреймворк для REST API |
| **База данных**              | PostgreSQL                                    | Хранение пользователей, файлов, версий и ссылок |
| **ORM / Доступ к БД**        | [GORM](https://gorm.io/)                      | Миграции и связи между таблицами |
| **Объектное хранилище**      | S3 / MinIO                                    | Хранение файлов (до 10 GB). MinIO для локальной разработки, S3 — продакшен |
| **Очередь / Брокер сообщений** | Apache Kafka                                  | Асинхронная обработка задач (генерация превью, истечение ссылок) |
| **Аутентификация / Magic Link** | JWT / кастомный flow с magic link           | Безпарольная аутентификация через временные ссылки |
| **Фоновые воркеры**          | Горутины + Kafka consumer'ы                   | Масштабируемая асинхронная обработка задач |
| **Конфигурация / Env**       | Viper / Envconfig                             | Управление конфигурацией через env и файлы |
| **Логирование / Мониторинг** | Logrus / Zap                                  | Логи для отладки и мониторинга |
| **Тестирование**             | Go test, Testify                              | Unit и интеграционные тесты |
| **Контейнеризация / Деплой** | Docker / Docker Compose                       | Лёгкая локальная разработка и деплой |
| **CI/CD**                    | GitHub Actions / GitLab CI                    | Автоматическая сборка, тесты, деплой |

> 💡 **Заметки:**
> - **Gin + GORM** — лёгкий и стандартный стек для Go.
> - **Kafka** позволяет масштабировать фоновые воркеры и обрабатывать большие объёмы задач асинхронно.
> - **MinIO** используется для локальной разработки вместо S3.
> - Воркеры реализованы с использованием горутин и Kafka‑консьюмеров, чтобы API оставался отзывчивым.

## Архитектура

Система реализована как **монолитное приложение** с использованием слоевой (Layered) и гексагональной (Hexagonal) архитектуры. Это даёт чёткое разделение обязанностей, тестируемость и простоту сопровождения при одновременном упрощении деплоя.

**Структура слоёв:**
- **Presentation / API Layer** — обработка HTTP‑запросов, magic‑link, загрузка/скачивание файлов, генерация публичных ссылок.
- **Application / Service Layer** — координация операций с файлами, версионирование, управление ссылками и фоновые задачи.
- **Domain / Core Layer** — бизнес‑логика и сущности (User, File, FileVersion, MagicLink, PublicLink).
- **Infrastructure Layer** — конкретные реализации внешних зависимостей: БД (Postgres), объектное хранилище (S3/MinIO), очередь задач (Kafka).

**Принципы Hexagonal (Ports & Adapters):**
- Domain‑слой независим от внешних сервисов.
- **Порты** определяют интерфейсы для доступа к БД, хранилищу и очереди.
- **Адаптеры** реализуют эти интерфейсы, что позволяет легко менять инфраструктуру без изменения бизнес‑логики.

**Фоновые воркеры:**
- **Preview Worker** — асинхронная генерация превью (потребляет задачи из очереди).
- **Public Link Expirer** — деактивация публичных ссылок по истечении TTL.
- Воркеры запускаются в составе монолита, но взаимодействуют через очередь для масштабирования и асинхронности.

> ⚠️ **Монолит vs Микросервисы:**
> Текущий подход — монолит, по следующим причинам:
> - **Простота** — код легче понимать и поддерживать в одном репозитории.
> - **Удобство деплоя** — деплоится один сервис; внешние зависимости (БД, S3, Kafka) остаются отдельно.
> - **Лёгкая локальная разработка и тестирование.**
> - **Масштабирование критических компонентов** (Preview Worker, Expirer) возможно через запуск нескольких процессов.
> - **Готовность к будущему разделению на сервисы**, благодаря слойной и гексагональной структуре.

## Безопасность / Аутентификация

Система использует **парольнуюless‑аутентификацию** (magic links) и набор мер защиты для файлов и публичных ссылок.

### Аутентификация

- **Magic Links (вход без пароля)**
  - Пользователь получает временную ссылку на email.
  - Ссылка содержит подписанный токен с TTL.
  - После подтверждения выдается сессионный JWT для API.
  - Пароли не сохраняются в БД.

- **Управление сессиями**
  - Сессии хранятся в БД с временем жизни.
  - API-запросы требуют сессионный токен в заголовке `Authorization`.
  - После истечения сессии пользователь запрашивает новый magic link.

### Авторизация

- **Доступ к файлам**
  - Пользователь видит только свои файлы или файлы, доступ к которым открыт через валидную публичную ссылку.
  - Ограничения обеспечиваются связями в БД и логикой приложения.

- **Публичные ссылки**
  - Временные ссылки с TTL ≤ 600 секунд.
  - **Public Link Expirer** автоматически деактивирует просроченные ссылки.
  - Доступ по публичной ссылке — только для чтения.

### Защита данных

- **Транспорт** — весь трафик API должен идти по HTTPS.
- **Хранение файлов** — S3/MinIO могут быть настроены с серверным шифрованием.
- **Чувствительные данные** — в БД хранится минимальный набор метаданных; паролей нет.

> 💡 **Заметки:**
> - Комбинация passwordless, TTL‑ссылок и защищённого транспорта обеспечивает удобство и защиту.
> - В дальнейшем можно добавить 2FA, шифрование файлов в покое и дополнительные механизмы управления токенами.

## API (Документация)

Система предоставляет RESTful API для управления пользователями, файлами, версиями, magic‑link и публичными ссылками. Все запросы и ответы — в JSON (кроме потоков файлов).

| Endpoint                                   | Метод | Описание                                         | Тело запроса / Параметры | Ответ |
|--------------------------------------------|--------|--------------------------------------------------|--------------------------|-------|
| `/auth/magic-link`                         | POST   | Запросить magic‑link для входа                   | `{ "email": "user@example.com" }` | `{ "magic_link": "https://..." }` |
| `/auth/magic-link/verify`                  | POST   | Подтвердить magic‑link и создать сессию          | `{ "token": "..." }` | `{ "session_id": "..." }` |
| `/files/upload`                            | POST   | Загрузить **новый файл** и создать первую версию | `{ "name": "file.pdf", "size": 1024 }` | `{ "file_id": 123, "upload_url": "https://..." }` |
| `/files/{file_id}/upload`                  | POST   | Загрузить **новую версию** файла                 | Path param: `file_id`, `{ "size": 2048 }` | `{ "version_id": 2, "upload_url": "https://..." }` |
| `/files/{file_id}`                         | GET    | Получить метаданные файла (последняя версия, владелец, имя, размер, created_at) | Path param: `file_id` | `{ "id": 123, "name": "...", "size": 1024, "owner_id": 1, "latest_version": 2, "created_at": "..." }` |
| `/files/{file_id}/rename`                  | PATCH  | Переименовать файл                                | Path param: `file_id`, `{ "name": "new_name.pdf" }` | `{ "status": "renamed", "new_name": "new_name.pdf" }` |
| `/files/download/{file_id}`                | GET    | Скачать **последнюю версию** файла               | Path param: `file_id` | File stream |
| `/files/{file_id}/versions`                | GET    | Список всех версий файла                          | Path param: `file_id` | `[ { "version_id": 1, "size": 1024, "created_at": "..." }, ... ]` |
| `/files/{file_id}/versions/{version_id}`   | GET    | Метаданные конкретной версии                      | Path param: `file_id`, `version_id` | `{ "version_id": 2, "size": 2048, "created_at": "..." }` |
| `/files/{file_id}/versions/{version_id}/restore` | POST | Восстановить версию как последнюю               | Path param: `file_id`, `version_id` | `{ "status": "restored" }` |
| `/files/{file_id}/versions/{version_id}`   | DELETE | Удалить конкретную версию                         | Path param: `file_id`, `version_id` | `{ "status": "deleted" }` |
| `/files/{file_id}/preview`                 | GET    | Получить превью для последней версии файла       | Path param: `file_id` | Image/Thumbnail stream |
| `/files/{file_id}`                         | DELETE | Удалить файл и все версии, включая превью и публичные ссылки | Path param: `file_id` | `{ "status": "deleted" }` |
| `/public-links`                            | POST   | Создать временную публичную ссылку для файла     | `{ "file_id": 123, "ttl": 600 }` | `{ "public_link": "https://..." }` |
| `/public-links/{link_id}`                  | GET    | Доступ к файлу по публичной ссылке               | Path param: `link_id` | File stream |
| `/public-links/{link_id}`                  | DELETE | Удалить публичную ссылку                          | Path param: `link_id` | `{ "status": "deleted" }` |
| `/users/me`                                | GET    | Профиль текущего пользователя                    | Auth token in header | `{ "id": 1, "email": "user@example.com", "files": [...] }` |

> 💡 **Заметки:**
> - **Версионирование:** Каждый вызов `/files/{file_id}/upload` создаёт новую запись `FileVersion`, а основная таблица `files` содержит метаданные по последней версии.
> - **Удаление:**
>   - `/files/{file_id}` удаляет основной файл, все версии, превью и публичные ссылки (`ON DELETE CASCADE`).
>   - `/files/{file_id}/versions/{version_id}` удаляет только указанную версию.
>   - `/public-links/{link_id}` позволяет вручную удалить одну публичную ссылку.
> - **Восстановление версии:** `/files/{file_id}/versions/{version_id}/restore` делает старую версию текущей.
> - **Генерация превью:** Превью генерируются асинхронно; после загрузки новой версии превью может быть временно недоступно.
> - **Публичные ссылки:** TTL ≤ 600 секунд; истечение обрабатывается воркером Expirer.
> - Все эндпоинты, кроме публичных ссылок, требуют аутентификации с сессионным JWT.

## Дальнейшие улучшения / Масштабирование

Сервис спроектирован как MVP монолита, но возможны следующие улучшения:

| Область                      | Возможные улучшения / Расширения                | Примечания |
|-----------------------------|--------------------------------------------------|-----------|
| **Асинхронные задачи**       | Добавить антивирусную проверку, уведомления (email/webhook), очистку временных файлов | Отдельные тяжёлые задачи стоит вынести в фоновые воркеры |
| **Хранение / CDN**          | Интеграция CDN для доставки файлов и превью     | Ускоряет скачивания для пользователей по всему миру |
| **Масштабирование БД**      | Read replicas, партиционирование, шардинг        | Для больших объёмов данных и низкой латентности |
| **Переход к микросервисам** | Разделение на сервисы (API, preview, link management) | Даст независимый deploy и масштабирование |
| **Поиск и индексирование**   | Full‑text поиск по метаданным и содержимому      | Улучшает UX при больших объёмах файлов |
| **Безопасность**            | Шифрование файлов в покое, 2FA, улучшенное управление токенами | Повышает защиту данных пользователей |
| **Наблюдаемость**           | Prometheus + Grafana для метрик и алертов        | Помогает держать uptime ≥ 99.5% |
| **Горизонтальное масштабирование воркеров** | Запуск множества экземпляров Preview Worker и Expirer | Обеспечит обработку очередей до 10k задач |

> 💡 **Заметки:**
> - Эти улучшения опциональны для MVP, но рекомендуются при росте нагрузки.
> - Архитектура (Layered + Hexagonal) облегчает рефакторинг в микросервисы.
