## Введение

**Проект**: Облачное хранилище файлов (Google Drive Lite)  
**Цель**: Разработка веб-сервиса для хранения, управления и обмена файлами с поддержкой версионности и генерации превью.

| Метрика                       | Символ           | Требование            |
| ----------------------------- | ---------------- | -------------------- |
| Доступность сервиса           | $Uptime$         | $Uptime \geq 99.5\%$ |
| Время восстановления (RTO)   | $RTO$            | $RTO \leq 1 \text{ ч}$ |
| Точка восстановления (RPO)    | $RPO$            | $RPO \leq 1 \text{ ч}$ |
| Задержка API                  | $Latency_{p95}$  | $\leq 200 \text{ мс}$ |
| Пропускная способность (загрузка) | $RPS_{upload}$   | $\leq 100$ |
| Пропускная способность (скачивание) | $RPS_{download}$ | $\leq 300$ |
| Одновременные пользователи    | $N_{active}$     | $\geq 1000$ |
| Максимальный размер файла     | $Size_{file}$    | $\leq 10 \text{ ГБ}$ |
| Общая емкость системы (MVP)   | $Size_{total}$   | $\geq 1 \text{ ТБ}$ |
| Время обработки превью        | $T_{preview}$    | $\leq 30 \text{ с}$ |
| Вместимость очереди           | $N_{queue}$      | $\leq 10^{4}$ |
| Успешность задач              | $P_{success}$    | $\geq 99\%$ |
| Время жизни публичной ссылки  | $TTL_{link}$     | $\leq 600 \text{ с}$ |

## Технологический стек

Система построена на основе **современной экосистемы Go**, объединяя высокопроизводительные веб-фреймворки, надежные базы данных, масштабируемые очереди и фоновые обработчики.

| Слой / Назначение             | Технология / Библиотека                        | Примечание / Использование |
|-------------------------------|-----------------------------------------------|----------------------------|
| **Язык программирования**     | Go (Golang)                                   | Основной язык backend, компилируется в один бинарный файл, подходит для высокопроизводительных сервисов |
| **Веб-фреймворк / Router**    | [Gin](https://github.com/gin-gonic/gin)       | Высокопроизводительный HTTP-фреймворк для REST API |
| **База данных**               | PostgreSQL                                    | Хранение пользователей, файлов, версий и ссылок |
| **ORM / Доступ к БД**         | [GORM](https://gorm.io/)                      | Популярный ORM для Go, поддерживает миграции и связи таблиц |
| **Хранилище объектов**        | S3 / MinIO                                    | Хранение файлов (до 10 ГБ), MinIO для локальной разработки, S3 для продакшена |
| **Очередь / Message Broker**  | Apache Kafka                                  | Асинхронная обработка задач (Preview Worker, Public Link Expirer) |
| **Аутентификация / Magic Link** | JWT / собственный flow magic link            | Вход без пароля через временные ссылки, токены хранятся в БД |
| **Фоновые обработчики**       | Go routines + Kafka consumers                 | Масштабируемая асинхронная обработка задач |
| **Конфигурация / Env**        | Viper / Envconfig                             | Управление настройками через переменные окружения |
| **Логирование / Мониторинг**  | Logrus / Zap                                  | Логи для отладки и мониторинга |
| **Тестирование / QA**         | Go test, Testify                              | Unit и интеграционные тесты |
| **Контейнеризация / Развертывание** | Docker / Docker Compose                     | Упрощает локальную разработку и продакшен |
| **CI/CD**                     | GitHub Actions / GitLab CI                     | Автоматическая сборка, тесты и деплой |

> 💡 **Примечания:**  
> - **Gin + GORM** — легкий и стандартный стек для Go веб-сервисов.  
> - **Kafka** позволяет масштабировать фоновые обработчики и обрабатывать большие объемы задач.  
> - **MinIO** используется для локальной разработки вместо S3.  
> - **Workers** используют Go routines и Kafka consumers для асинхронной работы и высокой отзывчивости API.

## Архитектура

**Архитектура проекта**: Облачное хранилище реализовано как **монолитное приложение** с использованием слоистой и гексагональной архитектуры.  

**Слоистая структура**:
- **Presentation / API Layer** – обработка HTTP-запросов, magic links, загрузка/скачивание файлов, генерация публичных ссылок.  
- **Application / Service Layer** – координация операций с файлами, версиями, ссылками и фоновыми задачами.  
- **Domain / Core Layer** – бизнес-логика и сущности (User, File, FileVersion, MagicLink, PublicLink).  
- **Infrastructure Layer** – конкретные реализации внешних зависимостей: база данных (Postgres), хранилище объектов (S3/MinIO), очередь задач (Redis/RabbitMQ/Kafka).

**Принципы Hexagonal (Ports & Adapters)**:
- **Domain Layer** независим от внешних сервисов.  
- **Ports** определяют интерфейсы для доступа к БД, хранилищу и очередям.  
- **Adapters** реализуют эти интерфейсы, что позволяет легко заменять компоненты инфраструктуры.

**Фоновые обработчики**:
- **Preview Worker** – асинхронная генерация превью файлов.  
- **Public Link Expirer** – деактивация публичных ссылок после окончания TTL.  
- Обработчики запускаются внутри монолита, но взаимодействуют через очередь задач.

> ⚠️ **Монолит vs Микросервисы:**  
> Хотя современные облачные системы часто используют **микросервисы**, проект реализован как **монолит** для практичности.  
> - Простота разработки и деплоя  
> - Легкость тестирования  
> - Масштабирование критических компонентов через фоновые обработчики  
> - Возможность перехода на микросервисы в будущем благодаря слоистой + гексагональной архитектуре

## Безопасность / Аутентификация

### Аутентификация

- **Magic Links (вход без пароля)**  
  - Временные ссылки по email  
  - Каждая ссылка содержит подписанный токен с TTL  
  - После проверки выдаётся **JWT** для API  
- **Управление сессиями**  
  - Сессии хранятся в базе с временем жизни  
  - Запросы API требуют валидный токен в `Authorization`

### Авторизация

- **Доступ к файлам**  
  - Только владельцы файлов или пользователи с действительной публичной ссылкой  
- **Публичные ссылки**  
  - TTL ≤ 600 секунд, Expirer деактивирует их автоматически  

### Защита данных

- HTTPS для API  
- Шифрование S3/MinIO (опционально)  
- Минимум метаданных в базе, пароли не хранятся

|--------------------------------------------|--------|-------------------------------------------------|---------------|-------|
